# BoltForge R&C Port

## Start

1. Read `AGENTS.md`.
2. Read the relevant sections of `docs/ratchet_clank_pc_port_plan.md`.
3. For public references, local tools, and recommended sources, read
   `references/source-map.md`.

## Rules

- Keep copyrighted game data, extracted assets, original binaries, and
  proprietary SDK material out of the repository.
- Use synthetic fixtures for committed tests.
- Keep real ISO, PCSX2, Wrench, or extracted-asset comparisons local-only.
- Prefer clean-room behavior specs, schemas, and tests over copied decompiled
  code.
- Treat Wrench code as GPL-3.0 unless a maintainer records a different legal
  decision for a specific dependency path.
- Keep the project native. Do not introduce a dynamic PS2 CPU, GS, VU, or SPU
  emulation layer as the main architecture.

## Implementation Heuristics

- Build importer and viewer tooling before gameplay systems unless the task is
  explicitly runtime-focused.
- Start with R&C1 and one known build unless the task requires multi-game
  support.
- Preserve source provenance in every manifest and decoded asset record.
- Parse binary formats with explicit endianness, bounds checks, and structured
  errors.
- Represent unknown fields and object classes explicitly so later agents can
  refine them.
- Add tests close to parser and schema changes. Use generated bytes and small
  synthetic fixtures.
- When behavior is learned from traces or disassembly, document it as a spec or
  test case instead of copying implementation code.

## Documentation Targets

- Put long-lived project rules in `AGENTS.md`.
- Put architecture, legal, file-format, or workflow details in `docs/`.
- Put machine-readable schemas in `data/schemas/` once that tree exists.
- Put local-only research outputs under ignored directories such as
  `local_assets/`, `asset_store/`, or `.import-cache/`.
